# 07 — Anti-Patterns

Что мы уже пробовали и поняли что **не надо**. Каждый пункт — с конкретным кейсом из прошлого.

## Данные и ETL

### ❌ Фильтровать мусор на SELECT-слое вместо фикса парсера
**Кейс**: `client_transactions` содержала 94 мусорных строки пагинации из 1430 (6.6%). Первая реакция: «добавим WHERE не в билде кэша». Правильно: жёсткий регекс `^[A-Z]\d{2}-\d+$` в `parse_tickets_table`. 1430 → 1336 чистых строк навсегда.
**Правило**: Фикси парсер, а не следствия.
**Коммит**: fixed in `sync_client_transactions.py` (2026-04-14)

### ❌ Доверять одному источнику выручки когда их несколько
**Кейс**: колонка «Последний платёж» в `clients.html` строилась только по `client_transactions` (тикеты). Клиенты на абонементах и saldo выглядели «потерянными» — их движения идут через `ListadoMovimientosSaldo` и `ListadoClientes_Bonos_Movimientos`. 44 ложных «потерянных».
**Правило**: Знать все источники → использовать агрегат (`ListadoClientesMasGasto`) или признавать что показываем подмножество.
**Решение**: `09_DECISIONS.md` #002 — переход на `activity_bucket` из `amount_7d/30d/90d`.

### ❌ Добавлять новую таблицу там где хватает поля
**Кейс**: была мысль «заведём отдельный ETL на saldo + bono». Правильно: использовать уже существующие `amount_7d/30d/90d` из `client_revenue` (они уже агрегируют всё) → 20 минут работы вместо 6 часов и нового API.
**Правило**: Сначала проверь что уже есть. Новая таблица — крайняя мера.

### ❌ Писать в Sheets без dry-run
**Правило**: Любой sync-скрипт сначала `python etl/X.py` (dry-run), смотришь output, только потом `--write`.

### ❌ Обновлять cached Sheets, а не источник
**Правило**: Если данные некорректны — фикси в `v7padel_db` и в ETL который в него пишет. В `v7padel_cache` ничего вручную не трогать.

### ❌ Парсер занятости пишет частичные данные для «сегодня-в-процессе»
**Кейс** (2026-04-16): 15.04 в `occupancy_daily` = 45 ч, Matchpoint UI и API показывают 35.5 ч. Причина: cron в 23:50 15.04 забрал данные пока в Matchpoint ещё висели неотменённые брони → записал 45 ч. К утру 16.04 брони схлопнулись, API стал отдавать 35.5 ч, но к тому моменту cron уже спал до след. суток.
**Правило**: Значения в `occupancy_daily` для последних N дней (N≥3) нестабильны — cron `scrape_occupancy.py --write` уже переписывает 90 дней, значит со временем «залечивается» автоматически. Но если нужна точная цифра прямо сейчас для свежего дня — дёрнуть парсер руками: `python etl/scrape_occupancy.py --write --from YYYY-MM-DD --to YYYY-MM-DD`.
**Решение будущего**: добавить пост-hoc ревалидацию (скрипт `scrape_occupancy_refresh_recent.py` раз в 2-3 часа для последних 3 дней) в `05_BACKLOG.md`.

## UI и фронт

### ❌ Кэш браузера ломает UI после релиза
**Кейс**: пофиксил скрипт, зашёл — баг на месте. Потому что браузер взял старый закэшированный JS.
**Правило**: При любом UI-фиксе — бампни кэш-бустер `?v=k(N+1)` в подключениях.

### ❌ Hardcoded :root theme на одной странице
**Кейс**: `brain.html` имел свой цветовой набор, отличающийся от остального портала.
**Правило**: Единая тема. Если нужен вариант — через data-атрибут, не через отдельный `:root`.

### ❌ Отдельная страница-редирект
**Кейс**: `knowledge.html` — 1 строка `location.href='brain.html'`. Бесполезный файл, занимает место в sidebar и auth-guard.
**Правило**: Если нужен редирект — через sidebar-ссылку напрямую на целевую страницу.

## Деплой и домены

### ❌ Пытаться открыть `v7padel.pages.dev`
**Кейс**: сказал пользователю открыть `v7padel.pages.dev/clients.html` — не открылось. Реальный домен — `v7padel.space` (GitHub Pages с кастомным CNAME).
**Правило**: **Наш сайт — только `v7padel.space`**. Никаких pages.dev, vercel.app, прочих внутренних URL.

## Документация

### ❌ Длинные документы с копипастом чисел
**Кейс**: `V7_Padel_Brain.md` содержал числа которые отстали на 2 недели, плюс они же были в `CLAUDE.md`, плюс у меня в user-memory. 4-кратная фрагментация → всё врёт.
**Правило**: Numbers только в `04_CURRENT_STATE.md` (auto-gen). Все остальные доки ссылаются.

### ❌ База знаний без механизма обновления
**Кейс**: `V7_Padel_Brain.md` не обновлялся 2+ недели — устарел. Нет явного владельца, нет триггера обновления.
**Правило**: Doc-in-commit (см. `06_PRINCIPLES.md` #11) + авто-генерация live-частей + еженедельный ревью.

## Коммуникация

### ❌ Давать первую попавшуюся идею
**Кейс**: Володимир сказал прямо: «мне нужны продуманные предложения, а не первые попавшиеся». 
**Правило**: Если есть выбор — 2-3 варианта с pros/cons и рекомендацией. Не одна идея.

### ❌ Путать Олю-PM и Олю-CEO
**Кейс**: в какой-то момент база знаний могла её называть CEO. Она — **Project Manager, правая рука**.
**Правило**: См. `08_TEAM_AND_ROLES.md`. Оля — PM.

### ❌ Смешивать задачи 1 минуты и 10 часов
**Кейс**: бэклог 30+ пунктов без приоритетов и оценок → свалка, не план.
**Правило**: Все четыре измерения задачи: приоритет + тип + длительность + исполнитель. Max 5 в `NOW`.

## Бизнес-правила

### ❌ Давать скидки
**Правило**: Никаких. Объёмные пакеты и турнирные воучеры — часть продукта, не акции. См. `00_NORTH_STAR.md` → Позиционные правила.

### ❌ Считать выручку без исключения сотрудников
**Правило**: cid 14 (Olha Bo), 5, 7, 9, все `@v7padel.com` — исключать из клиентских отчётов. Их транзакции — служебные.

### ❌ Копировать расход из PnL в `fixed_opex.json` для прошлых месяцев
**Кейс**: хотели положить аренду 380 490 ₺ в json и считать из него везде.
**Правило**: `fixed_opex.json` — только для **live-месяца** (текущего). Прошлые месяцы → всегда из PnL через service account.

### ❌ Держать `memberships` в Sheets на ручном вводе
**Кейс**: Ivanna Rykova добавлена в Matchpoint 15.04.2026 (VIP), в Sheets — не было. Дашборд показывал 38 карт вместо 39, «Активные карты потолок 55» отставал.
**Правило**: `sync_memberships_matchpoint.py` ежедневно, полный rebuild из `ListadoAbonados.aspx`. Плюс отдельно считать «реально оплачено» (`paid_until ≥ today`) — в Matchpoint статус Active может висеть дольше подписки. См. ADR-010.

### ❌ Считать абонементы по status=Active без проверки paid_until
**Кейс**: 6 из 39 карт формально Active, но подписка истекла. Метрика «Активные карты» завышена, потолок 55 ближе чем кажется.
**Правило**: Для живых метрик — `status=Active AND paid_until ≥ today`. `memberships_total_paid` в `dashboard_kpis` — точная цифра.
