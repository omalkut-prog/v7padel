# 14 · Code Review — процесс

> **Периодичность**: раз в 2 месяца.
> **Длительность**: 2-3 часа (с артефактом + фиксы P0 на месте).
> **Установлено**: 2026-04-18 после первого ревью.

## Что делаем

Раз в 2 месяца провожу полный скан кодовой базы: backend ETL, frontend HTML/JS/CSS, data integrity, docs. Собираю находки, классифицирую по severity (P0/P1/P2), чиню P0 немедленно, P1 фиксирую в backlog, P2 — в anti-patterns.

## Зачем

Технический долг копится быстрее чем кажется. Последние 2 недели до первого ревью накопили:
- sync_customers выпал из cron на неделю (никто не заметил).
- Я дважды ломал /revenue (ReferenceError).
- 183 строки deprecated кода остались в dashboard.html.
- 8 страниц с устаревшим `data.js?v=` (не подхватывали новые helpers).
- Orphan таб `rev_weekly` писался в кеш, но никто не читал.

Всё это ловится за 2 часа раз в 2 месяца — в 3 раза дешевле чем разбираться с последствиями по одному.

## Инструменты

### Автоматика: `python etl/audit_tools.py`

Отчёт за 30 секунд по 9 проверкам:

| Check | Severity | Что отлавливает |
|---|---|---|
| `syncs` | P0 | Все ли `sync_*.py` и `scrape_*.py` в `run_all_etl.py::STEPS` |
| `cache_busters` | P1 | Версии `data.js?v=` и `theme.css?v=` согласованы между страницами |
| `dead_py` | P2 | Функции в `build_cache.py` определены но не вызываются |
| `duplicate_js` | P1 | Одноимённые `render*/init*` функции в разных HTML (копипаст) |
| `ui_try_catch` | P2 | Render-функции >15 строк без `try/catch` (UI fragile) |
| `todos` | P2 | `TODO/FIXME/XXX/HACK` без даты |
| `secrets` | P0 | Подозрение на хардкод password/token/api_key |
| `orphan_cache` | P2 | Табы в `v7padel_cache` которые фронт не читает |
| `stale_logs` | P1 | `sync-*.log` не обновлялся >7 дней |

Запуск:
```bash
python etl/audit_tools.py              # все проверки, pretty print
python etl/audit_tools.py --json       # машиночитаемый
python etl/audit_tools.py --check syncs # одна проверка
```

### Ручной скан (что автоматика не видит)

- Размеры HTML-файлов — подозрительно маленькие (<3KB = заглушка) или большие (>100KB = возможно можно разделить).
- Работоспособность страниц на мобиле (пройтись по topnav: dashboard, clients, revenue, finance, management, team).
- Консистентность ADR с кодом (ADR говорит X, код делает Y).
- Свежесть `04_CURRENT_STATE.md` (авто-генерится, но если cron не отработал — устарело).
- `intensive.html` / подобные «тяжёлые» страницы — что они и нужны ли.

## Классификация severity

**🔴 P0 — чиню в том же ревью:**
- Secrets в коде.
- Критичный sync не в cron (данные не обновляются).
- Мёртвый deprecated код в prod (лишний вес, запутывание).
- Устаревшие cache-busters → пользователи не получают фиксы.

**🟡 P1 — в `05_BACKLOG.md` с effort:**
- Дубли кода между страницами (рефакторинг нужен, но код работает).
- Файлы-заглушки без решения «наполнять или удалить».
- Архитектурные вещи требующие часов (>1ч) работы.
- Orphan данные (табы, функции) где польза удаления мала.

**🔵 P2 — в `07_ANTI_PATTERNS.md` или живём:**
- Stylistic / convention issues (try/catch на мелких функциях).
- TODO без даты в служебных скриптах.
- Дубли функций с одинаковым именем но разной логикой (не проблема).
- Self-references audit_tools.

## Артефакт

После каждого ревью — файл `site/docs/reviews/REVIEW_YYYY-MM-DD.md` по шаблону:

```markdown
# Code Review · YYYY-MM-DD

## Резюме (P0/P1/P2 counts + общий вывод)

## 🔴 P0 (чиню сейчас)
- [x] P0-1 ...
- [x] P0-2 ...

## 🟡 P1 (в BACKLOG)
- P1-1 ...
- P1-2 ...

## 🔵 P2 (в ANTI_PATTERNS или живём)
- P2-1 ...

## ✅ OK (проверено)
- ...

## План действий + следующий запуск (YYYY-MM-DD)
```

## Метрика процесса

Следить чтобы **число P0 падало** от ревью к ревью. Идеал — через 2-3 цикла 0 P0 на входе, всё ловится на стадии разработки.

Если P0 растут — значит процесс разработки неправильный (не запускаем `audit_tools.py` перед пушем большого changeset'а). Добавляем в pre-commit.
