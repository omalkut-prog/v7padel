# 15 · Сверка с ручным финучётом (финика)

> **Статус**: живой документ. Обновлять после каждой ежемесячной сверки.
> Создан: 2026-04-22.

## Зачем

У нас есть **параллельный** ручной учёт, который ведёт финансист:
**[Google Sheet финики](https://docs.google.com/spreadsheets/d/1jUxfbpHAWILhl6CV2I3wLBP0NlzeT2JPs97QmUyIyTg/edit)** (ID: `1jUxfbpHAWILhl6CV2I3wLBP0NlzeT2JPs97QmUyIyTg`).

Этот шит **НЕ ЯВЛЯЕТСЯ источником данных** для нашей БД (ADR-015: dedup, cash-basis). Он используется **только как сверочный слой** — раз в месяц сравниваем итоги, ловим расхождения, чиним категоризацию в нашем pipeline или в шите у фина.

## Структура шита финики

| Вкладка | Содержание |
|---|---|
| `Вспомагательный лист 1` | Дневной реестр доходов: дата, категория, подкатегория, метод платежа, валюта. Есть time-of-day разбиение (07-11/17-23 vs 11-17) и per court vs per person. |
| `вспомогательный лист 2` | Дневные остатки кассы по валютам (TRY/USD/EUR), инкассация, смена админов. |
| `Товары` | Продажа/себестоимость по группам: ракетки, мячи, футболки, полотенца, напитки, намотки. |
| `Остатки` | Остатки на счёте по валютам + выгрузка CRM. |
| `Сводные расходы по периоду` | Автосводка расходов: Персонал, Налоги, Админ, Банковские, Маркетинг, Товары, Инвентарь, Коммунальные, Инвестиции, Открытие площадки. |
| `Сводные доходы по периоду` | Автосводка доходов по категориям. |
| `tournament cost calculation` | Себестоимость турниров (аренда + призы + мячи + подарки). |

## Процесс сверки (раз в месяц, 1-5 число следующего месяца)

1. Дождаться пока фин заполнит предыдущий месяц.
2. Открыть «Сводные доходы по периоду» и «Сводные расходы по периоду» → поставить нужный месяц в B2.
3. Сравнить с нашими:
   - **Revenue по категориям**: `transactions.category` + `manual_entries` (kind=income).
   - **Expense по категориям**: `expenses.category`.
4. Составить таблицу расхождений. Норма: ±3% по каждой крупной категории.
5. Разобрать причины:
   - **Мелкие Δ (1-3%)**: фин округляет или помечает дату по оплате vs мы по услуге — игнорируем.
   - **Крупные Δ (>5%)**: баг в нашем pipeline или пропуск у фина → фикс.
6. Записать результат в этот файл в секцию «История сверок».

## Почему НЕ делаем это автоматически

1. **Двойной учёт** (ADR-015): если подключить как источник — сломается dedup. Фин вводит данные вручную, мы — из Matchpoint. Одна транзакция попадёт дважды.
2. **Запаздывание**: фин заполняет раз в неделю-месяц. Наш pipeline ежедневный (cron 23:50).
3. **Разная cash-basis трактовка**: фин может списывать тренировку по дате занятия, у нас cash-basis (по дате оплаты).

Сверка ручная по дизайну — это **контрольная точка**, не ETL-источник.
