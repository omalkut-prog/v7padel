## Принципы и правила (не нарушать)

- **Cash-basis учёт**: выручка датой получения денег. Multi-day авансы (Camp, VIP-карты) одной суммой в день прихода кэша. См. `reference_accounting_rules.md`.
- **SSoT (Single Source of Truth)**: `transactions` cash = primary везде. PnL = только сверка на /finance. Если на 2 страницах разные цифры за 1 месяц — это бага. ADR-020.
- **Matchpoint cid +1 offset**: наш `customer_id` в БД смещён на -1 от MP URL ID. Для FichaCliente используй `gotoClient(N)` из листинга, а не `cid+1`. См. `reference_mp_cid_namespaces.md`.
- **TR-locale comma decimal**: Sheets API теряет запятую — `'31299,99'` приходит как `3129999`. Backend `_fix_tr_locale` делит на 100 при превышении порога. Если видишь миллионные суммы у клиента — это лосс запятой.
- **Сегменты клиентов** (по выручке за 30д): Ядро ≥10k₺, Под риском (0/30д ≥10k/90д), Растущие 2-10k, Разовые. См. `reference_client_segments.md`.
- **Ретеншн порог**: после 3 визитов — регулярный, после 5 — sticky.

# Правила учёта данных

## Cash-basis + anti-double-count
# Правила учёта выручки (cash-basis + anti-double-count)

**Источник правды**: `site/docs/13_ACCOUNTING_RULES.md` + ADR-015, ADR-016.

## Принцип признания: cash-basis
Выручка = дата получения денег (не дата оказания услуги). Причина: FX-риск лиры, приоритет скорости ротации TRY → USD. Следствие: Bob Camp 140k получен 16.04 → вся сумма на 16.04, хотя услуга 19-25.04. VIP-карта на год получена → вся сумма в день покупки. Multi-day услуги авансом — НЕ распределяем. Accrual отложен (условие перехода в ADR-016).

## Ключевое правило

Одна транзакция = один источник. Источники в `v7padel_db.transactions`:

- `old_inc` / `new_inc` → 1GDRZ Income (ручной ввод админа)
- `matchpoint_topup` → Matchpoint POS top-up (автомат)
- `manual_*` → `manual_entries` таб (Camp/Intensive, сторно, редкое)

## Зоны перекрытия и что делает система

1. **Top-up**: 1GDRZ ↔ Matchpoint. `sync_topup_to_transactions.py` автоматом **пропускает** Matchpoint-копию, если в 1GDRZ уже есть запись на тот же `(date, total)`. Dedup реализован 2026-04-17.

2. **Любая `kind=income` в `manual_entries`**: `merge_manual_entries` в `build_cache.py` автоматом **переводит в `info`**, если `(date, round(total, 2))` совпадает с уже имеющейся строкой в `transactions`. Не теряется — видна в отчётах, но не идёт в агрегаты. Счётчик `auto_downgraded` в stats.

## Правило ввода для админа

Перед `kind=income` в manual_entries:
1. Нет ли в 1GDRZ Income такого платежа? Если да → `kind=info`, не `income`.
2. Не проходил ли через Matchpoint POS? Если да → `kind=info`.
3. Уверен, что нет нигде → `kind=income`.

## Что считается НЕ перекрытием (свободно пиши `income`)

- Camp / Intensive — только в `manual_entries`.
- Внешние турниры с ручным сбором — только в `manual_entries`.
- Сторно Matchpoint — `kind=void`, не `income`.

## Иерархия источников по типу метрики

| Тип метрики | Где брать | Где показывать |
|-------------|-----------|----------------|
| Оперативная выручка (real-time) | `transactions` (cash) | /dashboard, /revenue, /clients, /may-goal, /risk |
| Бухгалтерская выручка (закрытый период) | `pnl_data` (бухгалтер) | /finance (как референс рядом с Tx) |
| Per-client выручка | `client_transactions` или агрегаты `client_revenue` | /clients |
| Per-month выручка | cache `rev_monthly_tx` | везде |
| Per-week выручка | cache `rev_weekly_income` + `rev_weekly_goods` | /revenue |

## Системное расхождение PnL vs Tx

PnL **всегда** меньше cash на 1.5%-15.5% (бухгалтер исключает internal/voids/etc.). Это **нормально**, но визуально подсвечиваем на /finance в колонке «Δ Tx-PnL» — Володимир и бухгалтер видят и подгоняют.

## Реализованные правила

- `/dashboard kpi-rev`: cash из `rev_monthly_tx` (раньше был PnL → расхождение с /revenue).
- `/finance таблица`: колонки Выручка PnL + Tx (cash) + Δ + OPEX + PnL.
- `/revenue Пульс`: cash из `rev_weekly_income` + `rev_weekly_goods`.
- RPCH: revenue (PnL для last_pnl_month) / hours (occupancy) — но `last_pnl_month` = последний ЗАКРЫТЫЙ месяц (см. ADR-019).

## Чего НЕ делать

- НЕ показывать PnL на /dashboard как «текущую выручку» — не совпадает с /revenue.
- НЕ дублировать суммирование транзакций на фронте — брать из cache.
- НЕ «исправлять» PnL в build_cache — это бухгалтерская правда.

## Чек-лист при добавлении новой revenue-метрики

1. Это **оперативная** метрика? → `transactions` (cash).
2. Это **отчёт за закрытый период**? → можно PnL, но рядом с Tx для сверки.
3. Это **per-month**? → используй cache `rev_monthly_tx`.
4. Уже есть похожая метрика где-то ещё? → проверь что числа сходятся!

### ✅ Правильно


## Matchpoint фиксы +/-
---
name: Matchpoint plus-minus корректировки
description: Админы фиксят ошибки в Matchpoint парой charge+credit note. Парсер с 16.04.2026 тянет обе стороны (Venta + Abonos). Manual void больше не нужен для типового кейса.
type: project
originSessionId: fa2245d6-eefa-4354-afdf-605937c98d6c
---
Админы в Matchpoint **часто** фиксят ошибочные проводки парой `charge + credit note`, затем правильный charge. Пример (Ivanna VIP 15.04.2026):
- A26-03796: +109 000 ₺ (ошибочные даты fee 10)
- A26-03797: −109 000 ₺ (credit note — сторно)
- A26-03805: +109 000 ₺ (правильная проводка fee 11)

Нетто = 109 000 ₺.

**Why:** Matchpoint не даёт редактировать проведённый тикет — только credit note + перевыпуск. Это нормальный admin workflow.

**Где что живёт в Matchpoint:**
- Charges (положительные) — `Facturacion/ListadoTickets_Venta.aspx`
- Credit notes (сторно) — `Facturacion/ListadoAbonos_Venta.aspx` (**отдельный листинг**, легко пропустить!)

**Как устроен парсер (с 16.04.2026):** `sync_client_transactions.py` делает **два pass**:
1. Venta → `is_credit_note=0`, total положительный
2. Abonos → `is_credit_note=1`, total инвертируется в минус (в Matchpoint листинге показан положительным)

SUM(total) по клиенту = нетто. См. ADR-007 в `docs/09_DECISIONS.md`.

**How to apply:**
- Если видишь расхождение между реальной оплатой и клиентской карточкой — сначала проверь `client_transactions` на `is_credit_note` и Abonos-листинг в Matchpoint UI (не только Venta).
- `manual_entries kind=void` **не пиши** если в Matchpoint есть credit note. Парсер уже учёл. Двойной void = двойное вычитание (баг).
- `manual_entries kind=income с cid` **не пиши** если оплата уже прошла через Matchpoint (charge или phantom). Даст дубль в клиентской карточке. Если запись нужна только для клубной выручки — ставь cid пустым.
- Листинг `ListadoClientesMasGasto.aspx` (пишет `client_revenue`) сам считает нетто — это Matchpoint-агрегат, с ним всё ок независимо от нашей логики.

## Отмены броней
# Отслеживание отмен броней (plan/fact часов)

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-017 + `13_ACCOUNTING_RULES.md` раздел «Plan vs Fact по часам корта».

## Ключевой факт

`occupancy_daily.hours` = **факт** (активные брони на момент cron), НЕ plan.
Настоящий `plan = fact + cancelled_hours`.

## Источники

- `etl/sync_cancellations_matchpoint.py` — тянет отменённые брони из Matchpoint `/Reservas/ListadoReservas.aspx` с `Estado=anulada`, экспорт в xlsx (без pagination), парсит → `v7padel_db.cancellations`.
- `etl/build_cache.py::compute_occupancy_planfact` — merge `occupancy_daily` (fact) + `cancellations` (cancel) → `occupancy_planfact` таб с колонками `[date, weekday, plan_hours, cancelled_hours, fact_hours, pct_cancel, is_final]`.
- `compute_cancellations_dow` → таб `cancellations_dow` (% отмен по дням недели, 4 недели).

## UI

- `/revenue` секция 1 «Часы по дням месяца»: stacked bar (факт+отмены), в таблице колонка «Отмены». Tooltip показывает план/факт/% отмен.
- `/dashboard` пара карточек: «Отмены по дням недели» (bar по dow с цветами: красный ≥50%, жёлтый 35-50%, зелёный <35%) + «Вчера — план vs факт» (KPI финализированного дня).

## Наблюдения (на 2026-04-17)

- 1127 отменённых броней в окне −60..+60 дней.
- Суббота — **65% отмен** (худший день недели, массовые Americano/тренировки).
- Воскресенье — 59%.
- Вторник — 34% (лучший).
- 15.04 (ср): plan=77ч, fact=35.5ч, cancel=41.5ч (54% отмен).

## Что отмены НЕ меняют

- **Не** вычитают выручку автоматически. Если charged=X, деньги остались (cash-basis). Финансовая сторона — `manual_entries` kind=void или Matchpoint credit notes.

## Что отмены МЕНЯЮТ

- Операционную метрику «загрузка кортов». plan/fact/cancelled часы по дням и по дням недели. Это то, что нужно пользователю для принятия решений (загрузить слоты, договориться с клиентами, изменить политику no-show).

## Ограничения

- Колонка `Cancel.` в ListadoReservas — пометка 'X', но **нет даты отмены**. Нельзя анализировать «late cancellations» без дополнительного источника.
- `fact` для будущих дней = текущий snapshot (ожидаемый факт); если отменят ещё — fact станет меньше.
- Данные ретроспективные с февраля 2026 (глубина occupancy_daily).

## Per-court KPIs
# Per-court KPIs (главная индустриальная метрика падел-клуба)

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-018 + `13_ACCOUNTING_RULES.md` раздел про per-court.

## Формулы (запомнить наизусть)

```
RPCH = revenue(last_pnl_month) / hours(того же месяца, occupancy)
Revenue per court = total_revenue / 4
OPEX per court = total_opex / 4
P&L per court = (revenue - opex) / 4
Estimated revenue корта X = hours[X] × RPCH (пропорциональная оценка)
```

## Где смотреть

- `/dashboard` — 2 KPI карточки сверху + кликабельная модалка с per-court breakdown.
- `/revenue` Пульс — те же 2 карточки.

## Источники

- Revenue: бухгалтерский PnL за last_pnl_month.
- Hours total: occupancy_daily.hours за тот же месяц.
- Hours по кортам: bookings.duration_min где court_id=X, status=active.

## Что НЕ делаем

- Не пытаемся точно атрибутировать revenue к конкретному корту (нельзя из-за тренировок/абонементов/турниров).
- Не считаем per-court RPCH — даёт ложную точность.
- Не делим OPEX по типу корта (P4 без крыши — пока поровну).

## Расписание

Считается **каждый день** в `etl/run_all_etl.py::build_cache` (шаг 15). Никакой отдельной обработки не нужно.

## История

- 2026-04-18: запрошено пользователем как «чуть ли не самая главная метрика в падел клубах».
- Первая попытка считала per-court RPCH из bookings.amount — ошибочно (P1=2020, P2=905), смешивала разные типы броней. Пользователь указал.
- Исправлено: общий RPCH × часы корта = пропорциональная оценка. Цифры выровнялись.

## Текущие цифры — НЕ хранить в memory

Живые числа (RPCH, выручка/корт) смотреть в `dashboard_kpis` cache или на /dashboard. В memory не дублировать — устаревает при каждом обновлении PnL/transactions.

Бенчмарк падел-индустрии (для контекста): RPCH 2000-2500 ₺/ч у премиум-клубов в Анталии.

## Грабли (ADR-019, 2026-05-03) — НЕ повторять

1. **`occupancy_daily` дубликаты** → RPCH занижен в 2x. Дедуп встроен в `scrape_occupancy.py` (last wins). Если RPCH резко упал — первое место смотреть.
2. **`last_pnl_month` ловил live-месяц** (текущий N-дневный) → RPCH ≈ 1000. Фикс: `closed_months = [m for m in pnl if m < cur_ym]`. Sanity check встроен.
3. **`/revenue.html ₺/ч` использовал только income** (без goods). Теперь income+goods. UI показывает ⚠️ если periodPerH < 500 при periodHours > 50.

Если `revenue_last_month` в `dashboard_kpis` ≈ 100K вместо ~1.5M — это live-месяц залез вместо closed. Если `total_hours_lm` 2x от ожидания — дубликаты в occupancy_daily.

description: Аналитика V7 — после 3 визитов клиент начинает играть регулярно, после 5 — становится устойчивым (sticky). Используется в Q2 целях.

По данным V7 Padel:

**Применение**:
- В стратегии recall: возврат клиента без доведения до 5+ визитов = низкая отдача
- Q2 2026 Цель 2: 60% возвращённых из `31_90d` должны сделать ≥5 визитов за квартал (см. site/docs/00_NORTH_STAR.md и ADR-006)
- При обсуждении любых retention-инициатив — этот порог опорный
- Для UI/UX: 1-й, 2-й, 3-й визит — критический онбординг; 4-5-й — закрепление

### N1.6. Lead Funnel + CAC tracking (КРИТИЧНО) — 4 часа

> Володимир: «не вижу замеров почем лидов получаем, сколько они стоят, сколько приносят, разработка воронки и как её отслеживать — а это наверное одно из самых главных».

**Принцип**: один CAC и одно ARPU **не существует** — у каждого канала / сегмента своя экономика. Турист CAC=2000₺/LTV=5000 ≠ Локал-VIP CAC=5000/LTV=200К.

**Сегменты для tracking** (минимум):
1. **Туристы EU/RU** (Instagram + Booking partnerships) — короткий цикл, низкий LTV, высокий объём
2. **Турниры/кэмпы** (rating-driven) — выручка-под-событие, CAC = % от tournament fee
3. **VIP / Inner Circle** (referrals + личный контакт) — длинный цикл, высочайший LTV
4. **Новички / Open Games / бесплатные тренировки** (top-of-funnel) — большой объём, низкий ROI на 1-й бронь, но через 14d retention превращаются в активных
5. **Дети / family** (родители платят) — отдельная воронка, разные точки входа (школы, детские мероприятия)

**Что мерить (per канал × сегмент)**:
- **Lead cost** (CPL) — сколько потратили на маркетинг / число лидов из источника
- **Conversion rate** — лидов → платящих клиентов (по segment)
- **Time to first booking** — сколько от lead до первой брони
- **14d retention** для каждой когорты (см. /cohort.html)
- **LTV proxy** — выручка от cohort за 90/180 дней
- **CAC payback period** — когда LTV покрыл CAC

**Реализация (поэтапно)**:
- **Phase 1** (1 час): UTM-структура для всех ссылок (utm_source/medium/campaign/content). Spec под Эрдема.
- **Phase 2** (1 час): добавить `source` поле в Matchpoint customer (manual at registration). Источник варианты: Instagram_Ads, Instagram_Organic, Tournament, Referral_VIP, Tourist_Booking, Walk-in, Other.
- **Phase 3** (1.5 часа): новая страница `/funnel.html`: воронка по сегментам, CPL, CAC, payback period. Берёт из customers.source + ad_spend.json + cohort retention.
- **Phase 4** (30 мин): автоматизация — webhook от Instagram Lead Form / WhatsApp на наш backend → автозаписать source.

**Зависит от**: Backend Foundation (NOW.5) для phase 4. Phase 1-3 можно сделать без backend.

**Эффект**: понимание что 50К/мес Ads окупается через 30 дней по туристам и 90 дней по локалам, а на детей вообще не работает (например). Перераспределяешь бюджет осознанно, не «по интуиции».

### N1.11. AI Admin Agent (Phase 1) — natural language search для админа (1-2 дня)

> Володимир: «дай мне игроков на турнир американо новичков пятница 20:00, дай ДР в ближайшую неделю, дай аналитику VIP/Club...»

**Главное конкурентное преимущество**: ни один padel-клуб в Турции (и почти никто в мире) не имеет admin-агента который понимает natural language и даёт actionable lists.

**Что делает Phase 1**:
- Admin/Володимир пишет в окне: «найди игроков на американо в пятницу 20:00 для новичков»
- Backend endpoint `POST /api/agent/query` отправляет в Claude API с structured context (наша БД)
- Возвращает: список клиентов + telephony + last_visit + любимый партнёр + draft messages на их языке (TR/EN/RU/UA)
- Admin ревьюит, корректирует, отправляет (Phase 1 НЕ автоматизирует отправку)

**Use cases (10 базовых)**:
1. Игроки для турнира (filter by level, time pattern, segment)
2. ДР в ближайшие 7-14 дней + suggestion
3. VIP/Club analytics (уровень, пол, время, ДР, last contact)
4. Cold/slow клиенты (>14 дней не были)
5. Кому давно не делали touch (>30д без contact_log)
6. Кандидаты в Inner Circle (top по выручке + active)
7. Recall candidates (bucket 31_90d)
8. Members истекают через 14 дней
9. Клиенты-новички которые перестали (14d retention отвал)
10. Кросс-фильтры (например «турки + advanced + играли в субботу»)

**Multilingual drafts**: agent пишет приглашения на языке клиента (поле `language` в customers + автоопределение по тексту прошлых сообщений).

**Tech**:
- Backend `/api/agent/query` endpoint
- Claude API с structured prompt (~5 KB context per query)
- UI в `/client.html` + новая консоль `/agent.html` (главный терминал админа)
- Cost ~$10/мес на 1000 запросов

**Зависит от**: backend (есть), customers/bookings/transactions (есть).

**НЕ зависит от**: WhatsApp templates (Phase 1 не отправляет, только показывает drafts), AI auto-notes (это другой level).

**Risk**: минимальный — клиент НЕ взаимодействует с AI напрямую. Только админ.

**Trigger для запуска**: после того как Володимир скажет ОК на Phase 1 (предложение принято).

### N1.10. AI Auto-Notes — заметки на основе истории игр (3-4 часа)

> Володимир: «мне нужно чтоб ии заметки делал исходя из того что уже отыграл, те из истории».

**Идея**: вместо того чтобы Володимир/менеджер вручную писал «играет по утрам», AI анализирует историю и **сам генерит заметки-инсайты** в карточке клиента:

```
🤖 Авто-инсайт (обновлено 2026-05-04):
• Играет регулярно — 8 визитов за 30 дней (ср. 2 в неделю)
• Любит утро (Пн, Ср) — 70% игр до 12:00
• Стандартный партнёр: Андрей Петров (cid=42, 12 совместных игр за 90д)
• Любимый корт: P2 (60% бронирований)
• Покупает энергетики Monster ~2× в неделю
• Тренировки: 4 за 30д с коучем Марией (Beginner → Intermediate progress)
• Last contact 12д назад — пора touch
```

**Реализация (2 варианта)**:

**Вариант A — Rule-based (быстрый, без AI)** — 1.5 часа:
- Backend endpoint `GET /api/client/{cid}/insights`
- Анализ из bookings/transactions: топ-партнёр, любимые часы, средний интервал между играми, тренды
- Шаблонные строки на основе threshold'ов
- Минусы: жёсткие шаблоны, не адаптивно

**Вариант B — Claude AI (умный, адаптивный)** — 3-4 часа:
- Backend endpoint `GET /api/client/{cid}/insights?ai=true`
- Собирает structured data → отправляет в Claude API с prompt'ом
- Claude генерит 3-5 человеческих заметок-инсайтов
- Кэшируется в SQLite (refresh раз в день)
- Cost: ~$0.001 на клиента × 1350 = ~$1.5/мес для всех
- Плюсы: адаптивно, может ловить тонкости (например «возможно скоро подтянется к VIP — 95% активности и растёт»)

**Рекомендация**: начать с **A** (быстро + видно ценность), потом мигрировать на **B** если Володимир захочет нюансов.

**Где показывать**:
- На `/client.html?cid=N` в блоке «Заметки» — секция «🤖 AI-анализ» НАД ручными заметками
- Auto-refresh раз в день (cron)
- Можно «save as note» если AI сделал особенно хороший инсайт

**Зависит от**:
- Backend `/api/client/{cid}` endpoint (уже есть)
- bookings.customer_id (для партнёров) — может потребоваться доработка ETL
- top_clients_revenue (категории покупок) — уже есть

**Trigger для запуска**: после того как накопится ≥30 заметок руками от Володимира — увидим какие паттерны люди реально пишут, AI обучим на этих стилях.

# 1. Никаких выдумок + психо-профиль обязательно виден

В сессии 2026-04-30: «очень мне важно чтоб ты ничего не выдумывал, чтоб я не попал в неудобную ситуацию». «я не увидел психологический профиль, но очень мне важно».

**Жёсткие правила**:
1. **Никаких фактов без источника** — каждый нетривиальный факт имеет inline-ссылку (URL + дата). Нет источника — поле пустое + open question.
2. **Маркировка INFERENCE** — любое утверждение выведенное из публичных следов (психо-профиль, мотиваторы, стиль) маркируется как INFERENCE с указанием evidence.
3. **Confidence indicator** на каждом блоке: HIGH (3+ источника или прямые цитаты) / MED (1-2 источника) / LOW (только самоанкета).
4. **Психо-профиль обязательно виден** — отдельная заметная секция:
   - Big Five приближение с evidence
   - Giver/Taker/Matcher (Grant) с evidence
   - Мотиваторы
   - Стиль коммуникации
   - Каждый пункт с confidence
5. **Нет данных = пишем «недостаточно данных, не использовать»**, не наполняем фантазиями.

**How to apply**:
- Никогда «возможно его триггерит X» без подтверждения — может стать неловкой ситуацией
- Никогда позицию по политическим/ценностным вопросам без прямого цитируемого подтверждения
- Лучше пустая секция с честным «не нашёл» чем заполненная гипотезами

# 2. Нелинейные факты ВСЕГДА с контекстом + inline source

## FILE: reference_accounting_rules.md

# Правила учёта выручки (cash-basis + anti-double-count)

**Источник правды**: `site/docs/13_ACCOUNTING_RULES.md` + ADR-015, ADR-016.

## Принцип признания: cash-basis
Выручка = дата получения денег (не дата оказания услуги). Причина: FX-риск лиры, приоритет скорости ротации TRY → USD. Следствие: Bob Camp 140k получен 16.04 → вся сумма на 16.04, хотя услуга 19-25.04. VIP-карта на год получена → вся сумма в день покупки. Multi-day услуги авансом — НЕ распределяем. Accrual отложен (условие перехода в ADR-016).

## Ключевое правило

Одна транзакция = один источник. Источники в `v7padel_db.transactions`:

- `old_inc` / `new_inc` → 1GDRZ Income (ручной ввод админа)
- `matchpoint_topup` → Matchpoint POS top-up (автомат)
- `manual_*` → `manual_entries` таб (Camp/Intensive, сторно, редкое)

## Зоны перекрытия и что делает система

1. **Top-up**: 1GDRZ ↔ Matchpoint. `sync_topup_to_transactions.py` автоматом **пропускает** Matchpoint-копию, если в 1GDRZ уже есть запись на тот же `(date, total)`. Dedup реализован 2026-04-17.

2. **Любая `kind=income` в `manual_entries`**: `merge_manual_entries` в `build_cache.py` автоматом **переводит в `info`**, если `(date, round(total, 2))` совпадает с уже имеющейся строкой в `transactions`. Не теряется — видна в отчётах, но не идёт в агрегаты. Счётчик `auto_downgraded` в stats.

## Правило ввода для админа

Перед `kind=income` в manual_entries:
1. Нет ли в 1GDRZ Income такого платежа? Если да → `kind=info`, не `income`.
2. Не проходил ли через Matchpoint POS? Если да → `kind=info`.
3. Уверен, что нет нигде → `kind=income`.

## Что считается НЕ перекрытием (свободно пиши `income`)

- Camp / Intensive — только в `manual_entries`.
- Внешние турниры с ручным сбором — только в `manual_entries`.
- Сторно Matchpoint — `kind=void`, не `income`.

## История

- 2026-04-17: обнаружено завышение 15.04 на 143 800 ₺ (+44%). Ivanna VIP 109k и top-up'ы учтены 2-3 раза. Две защиты + документация. Реальная среда = 325 290 ₺.

# Отслеживание отмен броней (plan/fact часов)

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-017 + `13_ACCOUNTING_RULES.md` раздел «Plan vs Fact по часам корта».

## Ключевой факт

`occupancy_daily.hours` = **факт** (активные брони на момент cron), НЕ plan.
Настоящий `plan = fact + cancelled_hours`.

## Источники

- `etl/sync_cancellations_matchpoint.py` — тянет отменённые брони из Matchpoint `/Reservas/ListadoReservas.aspx` с `Estado=anulada`, экспорт в xlsx (без pagination), парсит → `v7padel_db.cancellations`.
- `etl/build_cache.py::compute_occupancy_planfact` — merge `occupancy_daily` (fact) + `cancellations` (cancel) → `occupancy_planfact` таб с колонками `[date, weekday, plan_hours, cancelled_hours, fact_hours, pct_cancel, is_final]`.
- `compute_cancellations_dow` → таб `cancellations_dow` (% отмен по дням недели, 4 недели).

## UI

- `/revenue` секция 1 «Часы по дням месяца»: stacked bar (факт+отмены), в таблице колонка «Отмены». Tooltip показывает план/факт/% отмен.
- `/dashboard` пара карточек: «Отмены по дням недели» (bar по dow с цветами: красный ≥50%, жёлтый 35-50%, зелёный <35%) + «Вчера — план vs факт» (KPI финализированного дня).

## Наблюдения (на 2026-04-17)

- 1127 отменённых броней в окне −60..+60 дней.
- Суббота — **65% отмен** (худший день недели, массовые Americano/тренировки).
- Воскресенье — 59%.
- Вторник — 34% (лучший).
- 15.04 (ср): plan=77ч, fact=35.5ч, cancel=41.5ч (54% отмен).

## Что отмены НЕ меняют

- **Не** вычитают выручку автоматически. Если charged=X, деньги остались (cash-basis). Финансовая сторона — `manual_entries` kind=void или Matchpoint credit notes.

## Что отмены МЕНЯЮТ

- Операционную метрику «загрузка кортов». plan/fact/cancelled часы по дням и по дням недели. Это то, что нужно пользователю для принятия решений (загрузить слоты, договориться с клиентами, изменить политику no-show).

## Ограничения

- Колонка `Cancel.` в ListadoReservas — пометка 'X', но **нет даты отмены**. Нельзя анализировать «late cancellations» без дополнительного источника.
- `fact` для будущих дней = текущий snapshot (ожидаемый факт); если отменят ещё — fact станет меньше.
- Данные ретроспективные с февраля 2026 (глубина occupancy_daily).

# Балансы клиентов в Matchpoint — единственный правильный источник

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-022 (2026-05-03) + `etl/scrape_balances.py`.

## Правило одной строкой

> Балансы клиентов берём из **`/Clientes/ListadoClientesConSaldo.aspx`** одним запросом. **НЕ** перебираем 1350+ клиентов через `FichaCliente.aspx`.

## Где брать балансы

### ✅ Правильно

```
URL: https://v7padel.matchpoint.com.es/Clientes/ListadoClientesConSaldo.aspx
Метод: GET (page 1) → POST с __doPostBack для page 2+
Pagination target: ctl01$ctl00$CC$ContentPlaceHolderListado$GridViewListado
Pagination arg: Page$N
```

Возвращает таблицу: `Code | Name | 1st Last Name | 2nd Last Name | Mobile | Email | Membership Code | Balance`. Это в точности то что Володимир видит в **MP → Finance → Clients with Balance**.

**Скрипт**: `etl/scrape_balances.py --write` — записывает в `v7padel_db.client_balances`.

**Время**: ~5 секунд для всех страниц (~40 клиентов). Vs 30+ минут перебором.

### ❌ Не делать

- НЕ перебирать всех клиентов через `FichaCliente.aspx?id={cid}` для получения балансов — медленно (1350 запросов) и страдает от off-by-one (см. `reference_mp_cid_namespaces.md`).
- НЕ полагаться на `debts.balance` — там балансы только для клиентов которые попали в bookings последних N дней. Клиенты с балансом без недавних броней пропускаются.

## Структура `client_balances` tab

| Поле | Описание |
|------|----------|
| `customer_id` | display code parsed как int (000006 → "6") — совпадает с нашим `customers.customer_id` |
| `display_code` | оригинал из MP UI (например `000006`) |
| `name` | first_name + last_name |
| `first_name`, `last_name` | по отдельности |
| `mobile`, `email` | контакты |
| `membership` | код членства (например `/2026-2`) |
| `balance` | в TRY |
| `updated_at` | когда scraped |

## Cron расписание

ETL `run_all_etl.py` группа G4:
1. `scrape_debts.py --update --write --days 7` — должники (per-booking фокус)
2. `scrape_balances.py --write` — все балансы (одним запросом)

Запускается ежедневно 23:50.

## Где используется

| Файл | Что делает |
|------|-----------|
| `site/audit.html` Профицит таб | читает `client_balances` напрямую (не debts.balance) |
| `site/audit.html` KPI «Профицит балансов» | сумма `client_balances.balance > 0` |
| Будущий /clients финансовый блок | TBD — баланс клиента в его карточке |

## Связанные движения баланса

Отдельная таблица `balance_log` (тоже из `scrape_debts.py`) хранит **последние ~25 операций** по балансу:
- `Recharge` — пополнение баланса (клиент закидывает деньги наперёд)
- `Purchase` — списание (баланс использован для оплаты брони)
- `Correction` — ручная корректировка администратором

Поля: `date`, `type`, `client` (имя), `admin`, `description`, `amount`, `balance_before`, `balance_after`, `amount_num`.

**На /audit** при hover на имени клиента в Профицит — popup с этими движениями (через JS map по `client` имени, т.к. в `balance_log` нет `customer_id`, только имя).

## Грабли (не повторять)

- Раньше пытались через `FichaCliente.aspx` + off-by-one фикс — частично работало для low-cids, но пропускало клиентов с другим смещением.
- Был «Ibrahim Kurt 19 200 ₺» в /audit — это была неправильная attribution: его cid=1109 в нашей БД, но в MP он удалён, и url=cid+1=1110 возвращал баланс **другого** клиента (Sefa Seyrek 19 200). Финальный robust фикс — verify через `TextBoxCodigo` со страницы. Но **намного проще** просто использовать `ListadoClientesConSaldo.aspx`.

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-021 + `etl/build_cache.py:2198-2204` (компонент `compute_new_clients_retention`).

| Файл / функция | Использует cid в URL? | Применил +1? |
|----------------|----------------------|--------------|

| Будущие Telegram-боты для клиентов | если будут ходить по cid в URL — **обязательно +1** | TBD |

## Чек-лист при добавлении новой функциональности с MP cid

1. Это **массовая выгрузка** (ListadoClientes, customers tab)? → cid из БД работает напрямую.
2. Это **per-client URL** (`FichaCliente.aspx?id=`)? → **обязательно `cid + 1`** в URL.
3. Это **booking tooltip** (`ObtenerInformacionReservaTooltip`)? → **третий namespace**, нужен mapping bookings.cid ↔ tooltip.cid (см. `compute_new_clients_retention` решает через bookings без join'а).
4. **Тест** — взять несколько известных клиентов с balance/debt и сверить ожидаемое vs полученное.

## Историческая справка

- 2026-04: первый раз заметили проблему в `compute_new_clients_retention` — комментарий в коде упоминает «два namespace'а cid». Решение для retention: считать всё из `bookings` без join.
- 2026-05-03 (1): Володимир дал ground truth «у Рыкова 19 009 ₺» — наш fetch вернул 0. Прямой тест MP API подтвердил: `id=6 → balance 0`, `id=7 → balance 19009`. Применили `cid+1`.
- 2026-05-03 (2): Володимир указал что у Ibrahim Kurt 19 200 ₺ — но такого клиента в MP нет. Раскопали: `+1` НЕ универсален. Для **некоторых cids** сдвиг 0 или +/-1, для **других удалённых cids** url=cid+1 даёт чужой баланс.
- **Финальный robust фикс**: парсим `TextBoxCodigo` со страницы FichaCliente и **верифицируем что page.code == наш cid**. Если ни +1, ни =, ни -1 не дают совпадение → возвращаем None (не приписываем чужой баланс). Проверено на 8 known клиентах.

# Per-court KPIs (главная индустриальная метрика падел-клуба)

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-018 + `13_ACCOUNTING_RULES.md` раздел про per-court.

## Формулы (запомнить наизусть)

```
RPCH = revenue(last_pnl_month) / hours(того же месяца, occupancy)
Revenue per court = total_revenue / 4
OPEX per court = total_opex / 4
P&L per court = (revenue - opex) / 4
Estimated revenue корта X = hours[X] × RPCH (пропорциональная оценка)
```

## Где смотреть

- `/dashboard` — 2 KPI карточки сверху + кликабельная модалка с per-court breakdown.
- `/revenue` Пульс — те же 2 карточки.

## Источники

- Revenue: бухгалтерский PnL за last_pnl_month.
- Hours total: occupancy_daily.hours за тот же месяц.
- Hours по кортам: bookings.duration_min где court_id=X, status=active.

## Что НЕ делаем

- Не пытаемся точно атрибутировать revenue к конкретному корту (нельзя из-за тренировок/абонементов/турниров).
- Не считаем per-court RPCH — даёт ложную точность.
- Не делим OPEX по типу корта (P4 без крыши — пока поровну).

## Расписание

Считается **каждый день** в `etl/run_all_etl.py::build_cache` (шаг 15). Никакой отдельной обработки не нужно.

## История

- 2026-04-18: запрошено пользователем как «чуть ли не самая главная метрика в падел клубах».
- Первая попытка считала per-court RPCH из bookings.amount — ошибочно (P1=2020, P2=905), смешивала разные типы броней. Пользователь указал.
- Исправлено: общий RPCH × часы корта = пропорциональная оценка. Цифры выровнялись.

## Текущие цифры — НЕ хранить в memory

Живые числа (RPCH, выручка/корт) смотреть в `dashboard_kpis` cache или на /dashboard. В memory не дублировать — устаревает при каждом обновлении PnL/transactions.

Бенчмарк падел-индустрии (для контекста): RPCH 2000-2500 ₺/ч у премиум-клубов в Анталии.

## Грабли (ADR-019, 2026-05-03) — НЕ повторять

1. **`occupancy_daily` дубликаты** → RPCH занижен в 2x. Дедуп встроен в `scrape_occupancy.py` (last wins). Если RPCH резко упал — первое место смотреть.
2. **`last_pnl_month` ловил live-месяц** (текущий N-дневный) → RPCH ≈ 1000. Фикс: `closed_months = [m for m in pnl if m < cur_ym]`. Sanity check встроен.
3. **`/revenue.html ₺/ч` использовал только income** (без goods). Теперь income+goods. UI показывает ⚠️ если periodPerH < 500 при periodHours > 50.

Если `revenue_last_month` в `dashboard_kpis` ≈ 100K вместо ~1.5M — это live-месяц залез вместо closed. Если `total_hours_lm` 2x от ожидания — дубликаты в occupancy_daily.

# SSoT (Single Source of Truth) для выручки V7 Padel

**Источник правды**: `site/docs/09_DECISIONS.md` ADR-020 (2026-05-03).

## Один принцип

> Одна метрика = одно число везде. Если на /finance Март = 1.6M, на /revenue должен быть 1.6M, а не 1.64M.

## Иерархия источников по типу метрики

| Тип метрики | Где брать | Где показывать |
|-------------|-----------|----------------|
| Оперативная выручка (real-time) | `transactions` (cash) | /dashboard, /revenue, /clients, /may-goal, /risk |
| Бухгалтерская выручка (закрытый период) | `pnl_data` (бухгалтер) | /finance (как референс рядом с Tx) |
| Per-client выручка | `client_transactions` или агрегаты `client_revenue` | /clients |
| Per-month выручка | cache `rev_monthly_tx` | везде |
| Per-week выручка | cache `rev_weekly_income` + `rev_weekly_goods` | /revenue |

## Системное расхождение PnL vs Tx

PnL **всегда** меньше cash на 1.5%-15.5% (бухгалтер исключает internal/voids/etc.). Это **нормально**, но визуально подсвечиваем на /finance в колонке «Δ Tx-PnL» — Володимир и бухгалтер видят и подгоняют.

## Реализованные правила

- `/dashboard kpi-rev`: cash из `rev_monthly_tx` (раньше был PnL → расхождение с /revenue).
- `/finance таблица`: колонки Выручка PnL + Tx (cash) + Δ + OPEX + PnL.
- `/revenue Пульс`: cash из `rev_weekly_income` + `rev_weekly_goods`.
- RPCH: revenue (PnL для last_pnl_month) / hours (occupancy) — но `last_pnl_month` = последний ЗАКРЫТЫЙ месяц (см. ADR-019).

## Чего НЕ делать

- НЕ показывать PnL на /dashboard как «текущую выручку» — не совпадает с /revenue.
- НЕ дублировать суммирование транзакций на фронте — брать из cache.
- НЕ «исправлять» PnL в build_cache — это бухгалтерская правда.

## Чек-лист при добавлении новой revenue-метрики

1. Это **оперативная** метрика? → `transactions` (cash).
2. Это **отчёт за закрытый период**? → можно PnL, но рядом с Tx для сверки.
3. Это **per-month**? → используй cache `rev_monthly_tx`.
4. Уже есть похожая метрика где-то ещё? → проверь что числа сходятся!

## Чек-лист при добавлении нового блока на страницу с period filter

> Баг 2026-05-03: OPEX-вкладка на /revenue ИГНОРИРОВАЛА фильтр периода и всегда показывала последний месяц. Каждый новый блок должен реагировать.

1. Блок читает `_periodFrom` / `_periodTo` (или helper `inPeriod(d)` / `ymInPeriod(ym)`)?
2. Если есть `_rerenderAllAfterPeriodChange()` — добавлен ли блок туда?
3. Если данные monthly granularity — есть ли helper `ymInPeriod(ym)` для проверки пересечения месяца с периодом?
4. Если данные lazy-loaded (как OPEX через `_opexRendered`) — кэшируются ли raw rows в `window._xxxCache` чтобы re-render не дёргал сеть?

## История

- 2026-05-03: Володимир указал на расхождение Март /finance vs /revenue (1.6M vs 1.64M). Запустили reconciliation, выявили системное -1.5..15.5% PnL vs Tx. Принят SSoT принцип. Зафиксирован ADR-020.
- 2026-05-03 (позже): Володимир заметил что Сб 02.05 показывает 93К вместо 79К из источника 1GDRZ. Раскопали: ETL запускается раз в сутки (23:50), админы правят строки днём — наш `transactions` был устаревший на 14.5К. ## ETL freshness — правило

Если пользователь говорит «у меня в источнике X, а у тебя Y» → **первый шаг**: запустить `python etl/etl.py` + `python etl/build_cache.py` руками. Если совпало — это была stale data. Если нет — реальный баг в parser/aggregation.
