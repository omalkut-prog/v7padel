8 разделов правил + реальная спецификация колонок v7padel_db · обновлено 2026-04-09

1 · Языки

  • Сайт (CEO/Оля): русский — основной
  • Персонал (турки): турецкий + английский
  • Турецкий — строгая орфография: İ Ş Ğ Ü Ö Ç обязательно
  • Если не уверен в турецком — пометка нужна проверка носителем
  • Никакого транслита

2 · Данные и источники

  • P&L — только из pnl_monthly. Никогда из transactions или expenses
  • Названия колонок — точные, не угадывать
  • Fallback без пометки в UI — запрещён
  • Нет данных для блока = сообщи что добавить в ETL, не строй на заглушках
  • Нет данных в UI = явное сообщение с причиной, никогда пустой блок или вечный спиннер

Агрегация P&L из pnl_monthly

  • Revenue = строки где level=0, branch=revenue, article_name содержит «Доход от деятельности»
  • OPEX = строки где branch=expense, article_name содержит «Условно-постоянные расходы»
  • Op. P&L = Revenue − OPEX

Связь клиент ← → членство

  • В memberships нет customer_id — связь через holder (имя) ← → customers.name
  • Нормализация: trim().toLowerCase().replace(/\s+/g, ' ')
  • 39 из 39 holders совпадают с customers.name (100% linkage)

3 · Код

3.1
Перед написанием — проверь есть ли уже в codebase. Не дублируй.
3.2
Общий код (gviz парсер, утилиты) — только в assets/data.js.
3.3
Все константы и ID таблиц — только в шапке файла (V7.SHEET_ID в data.js).
3.4
Максимум 3 gviz запроса на страницу. Если нужно больше — объединяй в ETL (pre-aggregated sheets).
3.5
Не добавляй новые библиотеки без разрешения. Разрешённые: Chart.js, marked.js, Inter (шрифт).
3.6
Если решение больше 50 строк — остановись, скорее всего есть проще.
3.7
Не трогай файлы вне задачи без явного разрешения.

Технический долг

  • management.html: 6 gviz запросов — нужен pre-aggregated лист в ETL
  • finance.html: 5 gviz запросов — аналогично
  • bookings.customer_id пуст — блокер для воронки по играм, heatmap per client. Нужен ETL backfill.

4 · ETL

4.1
При изменении логики ETL — dry-run обязателен перед реальным запуском.
4.2
При обычном ежедневном прогоне — dry-run не нужен.
4.3
Не ломай базу — перед любым rebuild проверь что схема не изменилась.

5 · Качество — перед каждым пушем

5.1
Все блоки показывают реальные данные — не нули, не спиннер.
5.2
Мобилка: таблицы скроллятся горизонтально, кнопки минимум 44px, текст читаем без зума.
5.3
Существующие страницы не сломаны после изменений общих файлов.
5.4
Один коммит = одна задача с описанием что и зачем изменено.
5.5
Если блок показывает нули — это баг, не норма. Разберись до пуша.

6 · UI

  • Все тексты на русском (CEO/Оля интерфейс)
  • Персонал видит турецкий + английский
  • Если gviz не ответил за 10 секунд — кнопка «Обновить», не вечный спиннер
  • Нет данных = сообщение с причиной на нужном языке

7 · Порядок работы

7.1
Задачи выполняются строго в том порядке в котором написаны.
7.2
Если встретил блокер без тебя — принимаешь решение сам, фиксируешь в сообщении, продолжаешь.
7.3
Перед сдачей — самоаудит: code review + проверка данных + тест мобилки + отчёт.

8 · Архитектурные принципы

1
Не автоматизируй плохой процесс — убери его.
2
Один источник правды для каждой цифры.
3
Данные вводятся один раз.
4
Не оптимизируй то чего не должно существовать.
5
Простота важнее полноты — если можно проще, делай проще.
6
Каждое решение проверяется: работает ли это в сети из 10 клубов?
7
Всё измеряется. Если метрику нельзя посчитать — фича не считается готовой.
